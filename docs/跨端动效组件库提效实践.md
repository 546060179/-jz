## 一、项目概述

本项目聚焦短剧/流媒体 App 中设计与研发协作的行业共性痛点，构建了一条 **"Figma 设计稿 → AI 语义解析 → 多端可运行代码"** 的自动化管线，覆盖 UI 界面还原与动效开发两大核心场景。

项目包含两大核心成果：

1. **AI 代码生成管线**（像素对齐组）：通过"命名协议 + AI 语义解析"，从 Figma 设计稿自动生成 iOS / Android / H5 三端生产级代码，人工修改比例仅 3%-5%
2. **跨端动效组件库**（九州动效组）：以 Motion Design Tokens 为核心约束、AI 为生产引擎，完成 7 类核心动效的四端标准化实现，交互动效开发效率提升 93%

两者结合，实现 **设计风格定义 → AI 生成双端可用组件体系 → 生成包含节点设计稿 + 直接引用的动效库 → 前端界面实现** 的全链路自动化交付。

## 三、背景与痛点

### 3.1 设计-研发协作的"效率黑洞"

| 痛点 | 具体表现 | 影响 |
|------|--------|------|
| **设计稿还原度低** | 开发手动还原时，对间距、颜色、字体、交互状态理解不一致，还原度仅 70%-80% | 每个页面需 2-3 轮"走查-修改-验收"，耗时 1-2 天 |
| **多端重复开发** | iOS / Android / Web 三端各写一遍，缓动曲线、时长等参数人工对齐 | 每个动效多端适配需 4 小时，且容易不一致 |
| **设计系统落地难** | Figma 组件库和前端组件库分离，设计规范更新后开发需手动同步 | 双端组件库版本不一致，Design Token 沦为摆设 |
| **动效缺乏标准** | 动效参数"凭感觉"写，不同页面风格混乱 | 无法复用，新页面动效需重复实现 |
| **大量低价值工作** | 开发花大量时间"堆盒子"写样式；设计师花时间在标注、切图和验收 | 无法聚焦业务逻辑和体验创新 |

### 3.2 为什么需要用 AI 来解决

传统解决方案（Storybook、手动维护 Design Token、代码生成插件）存在根本性局限——它们只是"工具"，无法"理解"设计意图：

- **无法理解语义**：传统工具只能识别图层是"矩形"还是"文本"，但无法知道这是"可滚动的列表区域"还是"固定在底部的 Tab 栏"
- **无法处理复杂逻辑**：无法从图层命名中识别"下拉刷新"、"横向翻页"、"多态按钮"等交互行为
- **灵活性差**：任何设计稿的微小变更都需要手动同步更新代码或组件库

因此，我们需要一个能够"理解"设计语言和交互逻辑的 AI 大脑，作为"翻译官"和"代码生成器"，从根本上重塑协作流程。

### 3.3 动效在产品体验中的核心价值

- **品牌识别**：统一的动效语言（缓动曲线、时长尺度、转场风格）强化品牌心智，形成独特的"动态 DNA"
- **用户引导**：合理的动效（微光效、焦点高亮、渐进展开）降低认知负荷，帮助用户理解界面层级与交互反馈
- **情感连接**：细腻的趣味动效（成功态弹跳、失败态抖动）传递产品温度，提升用户留存与好感度

## 四、核心解决方案

### 4.1 整体架构：四阶段自动化管线

```
阶段一：设计风格定义
  设计师产出设计风格指南 + 3-5 个代表性 Figma 页面
        │
        ▼
阶段二：AI 生成双端可用组件体系
  AI 自动抽取 Figma Library（颜色/字体/组件库）
  + 生成 iOS/Android/Web 组件代码
        │
        ▼
阶段三：生成包含节点设计稿
  图层命名规范化（命名协议 + AI Layer Renamer 插件一键批量重命名）
  + 动效组件库（Motion Design Tokens 约束下的 AI 受控生成）
        │
        ▼
阶段四：前端界面实现
  AI 代码生成管线（MCP 拉取 → 描述构建 → 门禁检查 → 三端代码生成）
  + 动效组件一行代码接入
```

### 4.2 成果一：AI 代码生成管线（UI 界面还原）

#### 核心创新：命名即协议

不依赖 AI 直接"看图"生成代码（准确率低），而是设计一套轻量级图层命名规范作为"协议语言"，让 AI 通过解析命名来精确理解页面结构、控件类型、布局关系和交互行为。

**命名公式**：`前缀/名称.行为@交互`

| 前缀 | 含义 | 示例 | AI 映射结果 |
|------|------|------|-----------|
| `L/` | 布局区域 | `L/body.scrollY` | UIScrollView（竖向滚动） |
| `I/` | 组件实例 | `I/RankCard@tap` | 可复用 UIView 子类 + 点击手势 |
| `C/` | 控件 | `C/avatar` | UIImageView |
| `T/` | 文本 | `T/title` | UILabel |

#### AI 的四重角色

1. **命名语义解析器**：解析自定义命名公式，精确理解页面结构
2. **智能推断引擎**：对免命名节点（纯装饰容器），AI 根据属性自动推断类型。一个 85 节点的页面，设计师只需命名约 58 个核心节点（减少 32%）
3. **样式零配置提取器**：从 Figma MCP 返回的结构化数据中直接提取精确的颜色 RGBA、字体族、字号、行高、间距、圆角等属性，无需创建任何 Design Token
4. **三端代码生成器**：根据中间产物"UI 描述文档"（descriptor JSON），同时生成三个平台的可运行代码：
   - iOS：UIKit Swift 代码，含完整 Auto Layout 约束、交互逻辑和状态机
   - Android：Kotlin Fragment + XML 布局 + strings.xml 资源文件
   - H5：Vue 3 SFC 单文件组件 + i18n JSON 国际化文件

#### 关键设计决策

- **质量门禁**：代码生成前强制检查命名合规率（≥95%）、默认命名残留（=0），不达标不生成代码
- **中间产物可审阅**：先生成结构化的"UI 描述文档"（Markdown 格式），开发者可审阅、修正 AI 的理解，避免"黑盒"风险
- **管道式架构**：每个阶段职责单一，输入输出明确，便于独立测试和扩展
- **零学习成本工具化**：通过 AI Layer Renamer 插件将命名规范内化为自动工具，设计师无需记忆规则，一键完成规范化

#### 代码质量与生产可用性

| 质量维度 | 具体表现 |
|---------|--------|
| 像素级精确 | AI 通过 Figma MCP 读取精确数值，生成 UI 与设计稿视觉还原度达 **95% 以上** |
| 代码规范性 | 遵循项目编码规范（命名风格、文件组织、注释格式），Auto Layout 约束完整，不存在写死坐标 |
| 组件复用 | AI 识别到 Figma 组件实例后，调用项目中已有的 UI 组件，确保代码与设计系统一致性 |
| 交互逻辑完整 | 通过命名协议中的行为后缀（`.scrollY`、`@tap`），AI 生成包含滚动、点击、状态机等完整逻辑 |
| 人工修改范围 | 纯 UI 层面修改几乎为零，整体人工修改比例控制在 **3%-5%** |

| 评估维度 | 覆盖范围 | 说明 |
|---------|--------|------|
| UI 还原工作覆盖率 | **60%-70%** | 覆盖布局结构、静态 UI 元素渲染、基础交互绑定；剩余为复杂业务逻辑 |
| 纯 UI 层可用率 | **95%-97%** | 仅看 UI 视觉还原层面，AI 生成的代码几乎无需修改 |
| 三端一致性 | **100%** | 三端代码均从同一份 descriptor JSON 生成 |

### 4.3 成果二：跨端动效组件库

#### 核心创新：Motion Design Tokens 约束下的 AI 受控生成

区别于"AI 黑盒生成代码"的传统模式，核心创新是 **用标准化 Tokens 体系约束 AI 生成逻辑**，实现"意图声明 → 参数映射 → 多端生成"的自动化闭环。

构建 **"5 级 Timing × 5 种 Easing × 5 种 Intent"** 的三维标准化体系，将抽象的动效意图（如 enter/exit/emphasis）映射为可量化的参数（如 enter = 300ms + ease-out），形成统一的动效"语言字典"。

**跨端调用示例**：
- React：`intent="enter"`
- Android：`MotionIntent.ENTER`
- iOS：`.enter`
- 三端自动映射到相同的 **300ms + ease-out**，无需人工对齐

#### 动效分类与工具选型

| 动效类型 | 适用场景 | 核心工具 | 核心优势 |
|---------|---------|---------|---------|
| **交互动效** | 按钮反馈、页面转场、加载 Spinner 等 | Kiro（Claude Opus 4.6） | 代码级生成、精确可控、直接生产环境可用 |
| **氛围动效** | 启动页、装饰背景、弹窗氛围等 | 即梦 / Gemini / Lottie | 快速生成、视觉丰富、低成本 |

两者结合，从功能性交互到情绪氛围全覆盖。

#### 组件库核心能力

- **Motion Design Tokens 体系**：5 级 timing + 5 种 easing + 5 种 intent
- **高效接入**：18 个效果预设，一行代码接入
- **无障碍适配**：三级无障碍降级（full / reduced / none）自动适配系统偏好
- **动态时长测算**：`dynamicDuration` 根据元素尺寸自动推算时长
- **多元素编排**：`stagger` 多元素交错编排
- **物理效果**：Spring 物理动画

| **进剧&观看** | 沉浸丝滑观看，有趣互动 | 卡片到视频页面无过渡；互动元素僵硬，点击无 icon 反馈 |
| **付费解锁** | 快速抓取支付信息 | 付费页面缺少微动效；挽留弹窗可优化 |
| **看完&复访** | 续看一键直达 | 历史记录进度条到卡片样式变化无过渡 |

### 7.2 动效问题归类与优先级

| 分类 | 问题 | 优先级 |
|------|------|--------|
| **页面转场与衔接** | 卡片→视频剧情页、上次观看→播放页、收起过渡僵硬 | P0 |
| **控件交互反馈** | 播放控制、功能按钮（收藏/选集/分享）、当前播放状态 | P0 |
| **聚焦与引导** | 商品卡面/折扣金额、各弹窗 | P0 |
| **状态提示与体验** | 页面加载样式 | P1 |

**核心差异化优势**：
1. **原生移动端优先 + 三端覆盖**：竞品核心输出均为 Web 前端代码，本方案已实现 iOS / Android / H5 三端代码生成
2. **"命名协议"实现精确语义传递**：AI 不需要"猜"，而是"读"——这是高还原度和高代码质量的根本保障
3. **零额外成本、零学习门槛**：完全基于团队已有的 Kiro 和 Figma 基础设施

## 九、长期规划

### 短期（1-2 个月）

- 完善代码生成器对更多 UIKit 控件（UICollectionView、UITabBarController 等）的支持
- 新增 Spring 物理动画的原生端实现及流光动效
- 新增 Sequence 序列动画的组件层封装，支持声明式多步骤编排
- 将 Bash 脚本封装为内部 CLI 工具或 Kiro 插件，降低使用门槛
- 编写详细的《设计命名规范手册》和《开发接入指南》

### 中期（3-6 个月）

- 完善 Android / H5 生成器对复杂控件（RecyclerView、UICollectionView 等）的支持
- 引入视觉回归测试工具（Percy、BackstopJS），将 AI 生成的 UI 与设计稿自动比对，生成量化还原度报告
- 构建可视化动效编辑器（Web），设计师可直接调参预览、导出代码片段
- 接入 Figma 插件，设计稿中的动效标注自动生成组件库调用代码
- 补充 Lottie 集成方案，支持复杂动效文件与组件库混合使用

### 长期愿景

- 建设 Web 服务，开发人员只需粘贴 Figma 链接，即可在线预览 UI 描述文档并一键下载多端代码
- 建立"AI 训练闭环"：收集开发者对 AI 生成代码的修正数据，用于持续优化模型
- 探索反向同步能力：支持开发者在代码中修改组件属性（如主题色）自动反向同步到 Figma 设计系统，实现真正的"双向协作"
- 形成完整的"AI + 动效设计系统"工具链：设计师在 Figma 标注动效意图 → AI 自动生成四端代码 → 组件库渲染 → 性能监控闭环
- 探索 AI 根据用户行为数据自动优化动效参数（如根据设备性能自动调整 timing scale）
